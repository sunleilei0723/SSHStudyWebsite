数组 哈希表 一次遍历
01

题目

给定一个整数数组 nums 和一个整数目标值 target，请你在该数组中找出和为目标值 target 的那两个整数，并返回它们的数组下标。

你可以假设每种输入只会对应一个答案，并且不能使用同一个元素两次。
示例 1
输入：nums = [2,7,11,15], target = 9
输出：[0,1]
解释：因为 nums[0] + nums[1] == 9，返回 [0, 1]
示例 2
输入：nums = [3,2,4], target = 6
输出：[1,2]
示例 3
输入：nums = [3,3], target = 6
输出：[0,1]
2 ≤ nums.length ≤ 10⁴ -10⁹ ≤ nums[i] ≤ 10⁹ -10⁹ ≤ target ≤ 10⁹ 仅存在一个有效答案
List[int] twoSum(self, nums: List[int], target: int) → List[int]
02

题目拆解

输入
整数数组 nums，目标值 target
输出
两个下标 [i, j]，使 nums[i]+nums[j]=target
约束
i ≠ j，每个元素最多用一次
保证
恰好存在一组有效答案
本质归类
查找问题 — 对于每个元素 x，快速查找 target−x 是否存在
03

解题思路

暴力枚举
双重循环遍历所有数对 (i, j)，检查 nums[i] + nums[j] == target。思路简单，但时间复杂度高。
时间 O(n²) 空间 O(1)
哈希表一次遍历
遍历数组时，用哈希表记录已见过的 {数值→下标}。对每个元素，先查 complement = target − nums[i] 是否在表中，若在则直接返回，否则将当前元素加入表。
时间 O(n) 空间 O(n)
💡 核心洞察
将「查找补数」从 O(n) 降到 O(1)，是优化的关键。哈希表的本质是用空间换时间——牺牲 O(n) 额外空间，将每次查找从线性扫描变为常数查询。

易错要点

      

    

    

      

        

        
先查询再插入——必须先检查 complement 是否已在哈希表中，再插入当前元素。若先插入再查询，当 nums[i] × 2 == target 时会错误地匹配到自己。

      

      

        

        
返回顺序——题目要求返回 [较小下标, 较大下标]。由于我们先遍历到的是较小下标，存入哈希表后，后续找到时直接 [hash_map[complement], i] 即可保证顺序。

      

      

        

        
重复元素——当数组中有重复值时（如 [3,3]），哈希表只保存最新下标即可，因为先查后插保证了不会匹配自身，而题目保证恰好一个答案。

      

    

  


  


  
  

    

      07
      
扩展思考

      

    

    

      

        
如果数组已排序？

        
可以使用双指针法（左右指针向中间逼近），空间复杂度降为 O(1)。但排序本身需要 O(n log n) 时间，且会丢失原始下标信息，需要额外处理。

      

      

        
如果需要返回所有满足条件的数对？

        
哈希表需要存储每个值的所有下标列表，并在查询时过滤掉自身。时间复杂度取决于答案数量，最坏情况 O(n²)（如全部元素相同且都满足条件）。

      

      

        
三数之和怎么解？

        
排序后固定一个数，对剩余部分使用双指针。时间 O(n²)，空间 O(1)（不计排序）。这就是 LeetCode #15 三数之和的经典解法，是两数之和的自然延伸。
